# GitHub Profile Revamp — Design Spec

**Date:** 2026-06-26
**Status:** Draft, awaiting user review
**Repo:** `medsabbar/medsabbar` (the GitHub profile README repo)
**Owner:** Mohamed Sabbar (CTO at IBTIKAR-Technologies)

## 1. Goal

Replace the current cliché-laden profile README with a lean, distinctive, professionally-toned version that:

- Reads as a real human (CTO) wrote it, not a template
- Is factually accurate and not aspirational
- Loads reliably without depending on third-party services that have been dying (Heroku free tier, Cyclic.sh shutdown)
- Ages well — minimal decoration, no trending widgets

## 2. Non-Goals (Scope Guard)

Explicitly out of scope:

- A separate portfolio website or blog
- Auto-syncing from external content (essays, Spotify, WakaTime)
- A custom domain or GitHub Pages site
- Migrating or modifying any other repos
- Editing the GitHub profile sidebar metadata (name, bio, location, company, social links) — that is done in the GitHub UI, not in this repo
- Updating LinkedIn, Twitter, or other external profiles
- A pinned projects / curated repos section in the README (explicitly excluded by user decision)

## 3. Approach

**Approach B: README + self-hosted stats Actions**

Three files in the repo, no new top-level directories. The Actions workflow regenerates two SVG files daily and commits them back to the repo, so the README renders reliable committed assets rather than calling out to third-party hosts that have been sunsetting.

Rejected approaches:

- **A. Static-only README** — cheaper now, but the current README already uses third-party Heroku/Cyclic/Vercel endpoints that have been dying. Will break again.
- **C. Multi-page / website workflow** — scope creep. User asked for a profile revamp, not a website.

## 4. Architecture

```
medsabbar/
├── README.md                      # The profile rendered on github.com/medsabbar
├── assets/
│   └── banner.svg                 # Hand-crafted SVG hero banner
└── .github/workflows/
    └── stats.yml                  # GitHub Actions: regenerates stats SVGs every 24h
```

Generated by the workflow (committed to the repo, not regenerated each page load):

```
assets/
├── stats.svg                      # github-readme-stats card
└── streak.svg                     # github-readme-streak-stats card
```

Rationale:

- Banner as a separate file under `assets/` so it can be edited/swapped without touching Markdown.
- Workflow lives in `.github/workflows/` following GitHub conventions.
- Generated SVGs are committed to the repo, so the README renders correctly even if Actions is delayed, broken, or the workflow file itself is removed later.

### 5.1 What got removed (vs. current README)

| Removed | Reason |
|---|---|
| Typing SVG (`readme-typing-svg.herokuapp.com`) | Heroku-route, dated, every visitor has seen the same template |
| "🚀 Passionate Developer \| 💡 Innovation Driven \| 🎯 Solution Focused" | Pure cliché |
| `const medsabbar = { ... }` About Me code block | Universal template, recognized instantly |
| Shields badge wall (4 languages + 4 frontend + 4 backend + 4 databases + 4 cloud) | Cut to 5 inline code tokens. Tech stack is not a personality. |
| "🎯 Current Focus" emoji bullets | Replaced with 2 prose lines |
| Activity graph (`github-readme-activity-graph.vercel.app`) | Third migration in 2 years (Heroku → Cyclic → Vercel). Drop. |
| "💭 Quote of the Day" | Noise. |
| Profile view counter (komarev) | Past its peak. |
| "Thanks for visiting! 😊" | Empty calories. |
| "Let's Connect" social block | GitHub sidebar already shows it. |
| Pinned projects / curated repos section | Excluded by user decision. |

### 5.2 What got kept

- A stats card and a streak card — but self-hosted via Actions, not via third-party endpoints.
- A link to the company GitHub org (IBTIKAR-Technologies).
- A low-key "Currently building" line referencing Mauritanian e-government services without naming the specific platform (Houwiyeti / Khidmaty) — wording is intentionally low-key per user direction.

### 5.3 Voice and tone

- Direct, third-person, no hype words (no "passionate," "rockstar," "ninja," "10x," "seasoned," "veteran").
- Specifics over adjectives: company name, location (implicit via company), real stack, real current project.
- Confidence in the work without boast. The CTO title is stated, not performed.
- Markdown formatting is restrained: bold for the role/company line and the "Currently building" line, otherwise prose.

## 6. Banner SVG Spec

### 6.1 Dimensions and palette

- **Dimensions:** 1280×320 (standard GitHub banner ratio; renders well on retina and standard DPI)
- **Background:** `#0d1117` (GitHub dark mode canvas — matches the rest of the dark UI)
- **Foreground text:** `#e6edf3` (off-white)
- **Accent:** `#22d3ee` (cyan, used for thin rules and small highlights)
- **Font:** `JetBrains Mono, Menlo, Consolas, monospace` (font name is a hint; layout uses fixed text widths so the fallback chain holds)

### 6.2 Layout

Left-aligned primary block, centered rule, right-aligned tagline block. From top to bottom:

1. Large monospace text: `mohamed sabbar` (lowercase, matches the GitHub handle aesthetic)
2. Subdued smaller line: `@medsabbar` (the GitHub handle)
3. Centered 1px cyan rule, ~60% width
4. Tagline: `Building the web, one less-buggy app at a time.`
5. Tiny footer line: `Nouakchott, MR → Nouakchott, World` (subtle, references location)

### 6.3 Production rules

- Pure SVG, no rasters, no external font files, no JavaScript
- No external image references — must render correctly through GitHub's camo proxy
- Hand-authored (~150 lines of SVG), not generated by a tool
- All text uses fixed positioning so layout holds across font fallbacks

### 6.4 Alt text (used in the README image tag)

`Banner: mohamed sabbar, CTO at IBTIKAR-Technologies — "Building the web, one less-buggy app at a time."`

## 7. GitHub Actions Workflow

### 7.1 File: `.github/workflows/stats.yml`

Behavior:

1. Triggers on a daily cron at 06:00 UTC and on manual `workflow_dispatch`
2. Checks out the repo
3. Runs upstream Actions to generate two SVGs:
   - `assets/stats.svg` using the `anuraghazra/github-readme-stats` GitHub Action (runs the generator locally in the workflow, not the public Vercel endpoint). Theme: `tokyonight` to match the current README's visual identity. Includes private commits so the count isn't artificially low.
   - `assets/streak.svg` using `DenverCoder1/github-readme-streak-stats@v2` (has a first-class Actions path; the most reliable streak option). Theme: `tokyonight`.
4. Commits the SVGs back to the repo with `[skip ci]` in the commit message to avoid recursive triggers
5. Pushes using the default `GITHUB_TOKEN` with `contents: write` — no PAT required

### 7.2 Why self-hosted beats the current setup

| Current dependency | Status | Risk |
|---|---|---|
| `github-readme-streak-stats.herokuapp.com` | Heroku free tier sunsetting | Will break |
| `github-readme-activity-graph.vercel.app` | Heroku → Cyclic (shut down 2025) → Vercel | Third migration in 2 years, will break again |
| `github-readme-stats.vercel.app` | Maintainer notice: "best-effort, can be unreliable" | Already flaky |

Self-hosting via Actions means the SVGs are committed assets served from `raw.githubusercontent.com`. They can never break because of a third-party host.

### 7.3 Trade-offs

- Stats are at most 24 hours stale (vs. live with hosted endpoints). Acceptable for a profile.
- One extra workflow file to maintain.
- Workflow is visible (it lives in the public repo). Harmless.

### 7.4 Failure mode

If the upstream action breaks or changes, the existing committed SVGs keep rendering. The README does not break when the workflow breaks.

## 8. Testing and Verification

This is a documentation/asset change, not a code change. "Testing" is visual + structural.

### 8.1 Manual visual checks (done at implementation time)

1. Render the README on `github.com/medsabbar/medsabbar` and confirm:
   - Banner SVG renders at full size, no broken-image icon
   - All inline text wraps correctly at narrow widths (test at 400px viewport)
   - Stats and streak SVGs render (first run will produce placeholders; second run after a manual workflow trigger will produce real data)
2. Trigger the Actions workflow manually from the Actions tab; confirm both SVG files are written
3. View the README on a mobile-sized viewport
4. View with JavaScript disabled (sanity check — should be no-op since there is no JS)

### 8.2 Structural checks

1. No `http://` URLs — only `https://` or scheme-less references
2. Every image has alt text
3. No external JavaScript
4. Markdown passes `markdownlint` (will be set up at implementation time; standard linter for GitHub READMEs)
5. No `pinned projects` block, no curated-repos block, no `/medsabbar?tab=repositories` link in the README body
6. Total line count ≤ 45
7. No image, script, or stylesheet is loaded from a third-party host — every asset either ships in the repo (`./assets/*`) or doesn't load at all

### 8.3 Acceptance criteria for "done"

- [ ] Banner SVG renders correctly in light and dark GitHub themes
- [ ] All four sections (hero / about / stack / numbers) visible without horizontal scroll at 1024px viewport
- [ ] Actions workflow runs successfully on first manual trigger
- [ ] No clichés from the current README remain (Section 5.1 delta list is fully cleared)
- [ ] `markdownlint` passes
- [ ] No pinned projects section in the README
- [ ] Banner alt text is present and descriptive
